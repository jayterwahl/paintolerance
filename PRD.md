# Pain Tolerance — Product Requirements Document

## Vision

The internet is increasingly populated by bots, bad-faith reply guys, and low-effort sneering. Most of it is contentless — designed to provoke an emotional reaction rather than make an argument. The Pain Tolerance is a Chrome extension that inoculates Twitter/X users against this by simulating a flood of hostile replies on their own posts. Like the pain tolerance training, it tests whether you can master your instinctive emotional reaction to scorn — except instead of a poisoned needle, it's a poisoned timeline.

The extension injects fake, visually indistinguishable reply indicators and reply threads beneath the user's own tweets. The fake replies come from generated personas and are produced entirely offline via a combinatorial mad-libs engine. No API keys, no network calls, no external dependencies.

## Current implementation snapshot

Pain Tolerance is now a WXT-powered, TypeScript-based browser extension. There is no source `manifest.json`; WXT generates the manifest from `wxt.config.ts` and the files under `entrypoints/`.

Primary target remains Chrome Manifest V3. The package includes WXT Firefox scripts, but Chrome/X/Twitter is the validated product target until Firefox behavior is explicitly QA'd.

Current content-script matches:

- `*://twitter.com/*`
- `*://x.com/*`

Current extension permissions:

- `storage` — saves the user's handle, active/inactive state, and intensity setting in `browser.storage.sync`.

## Core experience

When the extension is active and configured with the user's handle, tweets authored by that handle on Twitter/X can appear to have received a wave of hostile, dismissive, or absurdly off-topic replies. The current implementation focuses on visually plausible compact reply previews below detected tweets, plus reply-count inflation.

Current behavior details:

- The popup collects the user's Twitter/X handle, active toggle, and intensity.
- The content script watches Twitter/X's SPA DOM with a `MutationObserver`.
- Matching tweets are detected by scanning `data-testid="tweet"` cells and checking the author text for the configured handle.
- Quoted tweets are skipped; the extension only targets standalone tweet cells.
- In thread/status views, non-terminal tweets in a same-author thread are skipped so the extension does not flood every tweet in a thread.
- The focal tweet on a `/status/<id>` page is currently skipped to avoid layout overlap with Twitter/X's expanded tweet UI.
- Injected replies are deterministic per tweet ID, but the count per injection is selected from the active intensity range.

The user learns, over time, to notice the pattern: these replies contain no real argumentation, no substance, no signal. They are pure noise. That recognition is the training.

## Architecture overview

### WXT project shell

WXT provides the extension build system, manifest generation, Vite bundling, and generated TypeScript support.

Important files:

- `package.json` — npm scripts and dev dependencies (`wxt`, `typescript`).
- `wxt.config.ts` — global generated-manifest configuration.
- `tsconfig.json` — extends `.wxt/tsconfig.json` generated by `wxt prepare`.
- `entrypoints/` — WXT extension entrypoints.
- `utils/` — TypeScript modules imported by entrypoints.
- `public/` — static assets copied into the built extension.
- `.aside/` — prototype/reference materials, old store assets, screenshots, and the legacy extension source.

Generated directories are ignored and should not be edited by hand:

- `.wxt/`
- `.output/`
- `node_modules/`

### Components

**1. WXT config (`wxt.config.ts`)**

Defines global manifest fields:

- name: `Pain Tolerance`
- description
- `storage` permission
- extension-page CSP: `script-src 'self'; object-src 'self'`
- icons and action default icons using `public/icon-16.png`, `public/icon-48.png`, and `public/icon-128.png`

WXT adds the popup and content script manifest entries based on entrypoint files.

**2. Twitter/X content script (`entrypoints/twitter.content.ts`)**

This is the main runtime module. It combines the observer, filtering logic, reply injection, virtual-scroll overlay, settings loading, and cleanup.

Key responsibilities:

- Declares the WXT content script with `defineContentScript`.
- Matches Twitter/X URLs.
- Loads settings from `browser.storage.sync` using WXT's `wxt/browser` wrapper.
- Watches storage changes from the popup.
- Observes the Twitter/X DOM for newly mounted tweet cells.
- Filters to the configured user's own standalone tweets.
- Skips quoted tweets, non-terminal thread continuations, and the focal status tweet.
- Generates fake replies through `PT_YAPPER.generateReplies(tweetId, count)`.
- Builds reply preview DOM programmatically using `document.createElement` and SVG icons.
- Inflates the visible reply count when the reply-count span is found.
- Marks processed tweets with `data-pt-yapped="true"`.
- Uses WXT's `ContentScriptContext` helpers for scroll listener registration, animation-frame scheduling, and invalidation cleanup.

The script does not run DOM or extension API code at module top level. Runtime code is inside the WXT `main(ctx)` function.

**3. Virtual-scroll overlay (`entrypoints/twitter.content.ts`)**

Twitter/X often renders profile/thread content inside virtualized absolute-positioned containers. Direct sibling injection can be reset or misplaced by React. The current implementation handles these cases with a fixed overlay:

- Creates `#pt-vs-overlay` as a fixed, pointer-events-none container outside the React subtree.
- Adds reply containers into the overlay with `pointer-events:auto`.
- Tracks each injected container by tweet ID in `vsMap`.
- Repositions containers via `getBoundingClientRect()` on scroll and DOM mutations.
- Clears stale overlay entries on SPA URL changes.

Normal timeline-style pages that are not virtualized use direct insertion after the tweet cell boundary.

**4. Selector map (`utils/selectors.ts`)**

A single-file abstraction mapping logical names to current Twitter/X DOM selectors. Isolates DOM-structure assumptions so selector changes can be patched in one place.

Current logical names include:

- `TWEET_CELL` — `[data-testid="tweet"]`
- `TWEET_TEXT` — `[data-testid="tweetText"]`
- `TWEET_AUTHOR` — `[data-testid="User-Name"]`
- `REPLY_BUTTON`, `RETWEET_BUTTON`, `LIKE_BUTTON`
- `REPLY_COUNT`
- `BACK_BUTTON`
- `TWEET_LINK`
- `TWEET_AVATAR`
- `TIMELINE`
- `TWEET_THREAD`
- `QUOTE_CONTAINER`
- `SOCIAL_CONTEXT`
- `MARKER_ATTR` — `data-pt-yapped`

**5. Yapper engine (`utils/yapper.ts`)**

The content generation core. Produces typed fake reply objects containing:

- archetype ID
- display name
- handle
- verified flag
- avatar data URL
- reply text
- timestamp string
- engagement metrics: likes, retweets, replies, views, bookmarks

Generation pipeline:

1. Seed a deterministic PRNG from the tweet ID.
2. Roll an archetype via weighted random selection.
3. Select a template from the archetype's template bank.
4. Fill `{slot}` placeholders from `PT_CORPUS.fillers`.
5. Generate identity fields.
6. Generate avatar through `PT_AVATARS.generate`.
7. Generate engagement metrics.
8. Generate a relative timestamp.

`generateReplies(tweetId, count)` is deterministic for a given tweet ID and reply index.

**6. Corpus (`utils/corpus.ts`)**

The static data bank. Contains archetype definitions, templates, slot fillers, and identity generation material. It is bundled into the extension at build time.

The corpus is intentionally offline-only: no AI calls, API calls, or remote content loading.

**7. Avatar generator (`utils/avatars.ts`)**

Generates profile-picture data URLs using a canvas. The fake handle is hashed to deterministically select:

- background color from a curated muted palette
- whether the avatar has a letter or is solid color
- first letter from display name
- font size and weight variation

Output is a `data:image/png;base64,...` URL ready for an `<img>` `src`.

**8. Popup UI (`entrypoints/popup/`)**

WXT popup entrypoint:

- `entrypoints/popup/index.html` — popup document.
- `entrypoints/popup/main.ts` — settings load/save logic.
- `entrypoints/popup/style.css` — popup styling.

Settings:

- **Handle input** — stored as normalized handle without leading `@`.
- **Active toggle** — enables/disables observer behavior.
- **Intensity select** — controls how many fake replies appear per tweet:
  - Mild: 3–5 replies
  - Medium: 6–10 replies
  - Unhinged: 11–20 replies
- **Status indicator** — prompts for handle, shows active status, or inactive status.

## Archetype system

Each fake reply is generated from one of seven persona archetypes. Archetypes use relative weights, not normalized percentages.

| Archetype | Weight | Description | Example output pattern |
|---|---:|---|---|
| Ratio Guy | 22 | One-word or two-word dismissals. Maximum contempt, minimum effort. | `ratio`, `L`, `nobody asked`, `L + {insult}` |
| Unhinged Reply Guy | 15 | Disproportionate emotional reaction. All-caps optional. Treats mild takes as personal attacks. | `I cannot BELIEVE you would say this. {overreaction}. Blocked.` |
| Pseudo-Intellectual | 12 | Sneering condescension disguised as insight. Uses `tell me you don't understand X` formats. | `tell me you don't understand {concept} without telling me you don't understand {concept}` |
| Sneering Dismissal | 18 | More prose-heavy contempt and personal dismissal. | `the confidence in this post is inversely proportional to its quality` |
| Concern Troll | 8 | Faux-sympathetic framing that's actually dismissive. | `I mean this in the nicest way but {backhanded_concern}` |
| Off-Topic Ranter | 10 | Ignores the original tweet and launches into an unrelated grievance. | `speaking of which, {unrelated_rant}` |
| One-Word Oracle | 5 | Single word, emoji, or very short judgment. Enigmatic and infuriating. | `no`, `wrong`, `mid`, `delete this`, `💀` |

The corpus intentionally excludes spam-flavored archetypes (e.g., crypto/OnlyFans/"link in bio" bots). Simulated replies should feel specific and hostile rather than promotional.

## Template and slot system

Templates are strings with named slots wrapped in curly braces. Slots are filled from independent arrays under `PT_CORPUS.fillers`.

Current slot categories:

- `{insult}` — generic dismissals such as `this ain't it`, `touch grass`, `certified clown behavior`.
- `{concept}` — pseudo-intellectual topics such as `market dynamics`, `game theory`, `systems thinking`.
- `{overreaction}` — disproportionate responses such as `my therapist is going to hear about this`.
- `{backhanded_concern}` — concern-troll phrases such as `maybe log off for a bit`.
- `{unrelated_rant}` — off-topic tangents such as QR-code menus, printers, tipping culture, etc.

Current corpus notes:

- Many templates have no slots; some have one slot; a few reuse a slot multiple times.
- There is no current `{coin}` or `{shill_phrase}` slot.
- Generated text is deterministic from tweet ID + reply index, but not semantically aware of the original tweet text.

## Identity generation (`utils/yapper.ts` + `utils/avatars.ts`)

### Display names

Generated from first-name/generic-name material plus optional last fragments:

- Human-ish names: `Jake`, `Mike`, `Alex`, `Jordan`, `Avery`, etc.
- Generic persona names: `truth`, `freedom`, `based`, `alpha`, `sigma`, etc.
- Last fragments: `Official`, `Real`, `Based`, `NotABot`, `TheTruth`, `Poster`, `Takes`, etc.

### Handles

Current handle patterns:

- `{first}{4digit}` — e.g. `jake8294`
- `{first}_{fragment}` — e.g. `truth_TheTruth`
- `{adj}{noun}{2digit}` — e.g. `basedtakes99`
- `{first}{year}` — e.g. `mike1997`

Handles are returned with a leading `@`.

### Verification indicator

Approximately 20% of generated identities receive a simple blue checkmark glyph. This is a visual flourish only and not intended to model verified identity credibility.

## Metrics generation

Fake replies receive randomized engagement metrics following rough power-law-ish distributions.

Current distributions:

- **Likes**
  - 70%: 0–2
  - 20%: 3–15
  - 8%: 16–50
  - 2%: 51–200
- **Retweets**
  - 85%: 0
  - 10%: 1–3
  - 5%: 4–10
- **Replies**
  - 80%: 0
  - 15%: 1–3
  - 5%: 4–8
- **Views**
  - 50%: 50–499
  - 30%: 500–1,999
  - 15%: 2,000–4,999
  - 5%: 5,000–19,999
- **Bookmarks**
  - 70%: 0
  - 20%: 1–3
  - 10%: 4–20
- **Timestamps**
  - 1–240 minutes, displayed as `Xm` or `Xh`
  - Uses a squared random curve so more replies appear recent

## DOM injection strategy

### Selector stability

Twitter/X uses `data-testid` attributes on key elements. These are more stable than hashed CSS class names but can still change. `utils/selectors.ts` is the intended patch point when Twitter/X DOM changes.

Known selectors used by the current implementation:

- `tweet` — individual tweet cell
- `tweetText` — tweet body
- `User-Name` — author info container
- `reply`, `retweet`, `like` — action buttons
- `app-text-transition-container` — used inside the reply-count selector
- `app-bar-back` — retained in the selector map, though thread view is currently detected by URL path
- `quoteTweet` — quoted tweet container
- `socialContext` — social/thread context label

### Injection points

**Normal-flow timeline cells**

When a tweet cell is not in a detected virtual-scroll context and is not on a thread/status view, the reply container is inserted directly after the tweet's cell boundary.

**Virtual-scroll/profile/thread contexts**

When the tweet is in a virtualized absolute-position context, or the URL is a `/status/` thread view, the reply container is mounted into the fixed overlay and visually positioned below the tweet's current bounding box.

**Thread/status limitation**

The current implementation does not insert full fake reply tweet cells into the Twitter/X reply list. It uses the same compact reply preview container everywhere. The focal tweet on `/status/<id>` pages is skipped to avoid overlap. Supporting realistic full thread replies remains future work.

### Re-render resilience

Twitter/X destroys and recreates DOM nodes frequently because of virtual scrolling, tab switches, and SPA navigation. Current mitigation:

1. Processed tweet elements receive `data-pt-yapped="true"`.
2. Deterministic reply generation uses the tweet ID, so recreated tweets get the same fake replies.
3. Normal-flow deduplication checks whether the next sibling is already a `.pt-reply-container`.
4. Overlay deduplication checks `vsMap` by tweet ID.
5. Overlay entries are removed when their source cell leaves the document.
6. Overlay entries are cleared when `location.href` changes.

## Chrome Web Store considerations

### Policy compliance

- The extension modifies only the visual appearance of the user's own Twitter/X feed.
- It does not scrape data, intercept requests, or communicate with external servers.
- No data collection: the only stored data is the user's own handle and preferences in `browser.storage.sync`.
- No remote code execution: all generation logic is bundled statically by WXT/Vite.
- The listing should clearly state that the extension adds fictional UI content for training purposes.

### Listing description (draft)

> The internet is full of bots and bad-faith reply guys trying to get a rise out of you. The Pain Tolerance helps you build immunity.
>
> This extension adds simulated hostile replies to your own tweets — mean, dismissive, off-topic, and completely fake. The replies come from generated personas and are produced entirely offline. No AI, no API keys, no data collection.
>
> Over time, you'll notice the pattern: contentless sneering, engagement bait, derailed conversations. Once you can spot it in simulation, you'll spot it in the wild.
>
> Named after the pain tolerance training — a test of whether you can master your instincts under pressure.

## Known limitations / future improvements

- **Thread/status views**: The focal status tweet is skipped and full-size fake reply tweet cells are not implemented. Future work could add a dedicated thread-view injection strategy that avoids overlap.
- **Visual fidelity**: Reply previews are built with custom DOM and inline styles, not cloned Twitter/X tweet cells. They are plausible but not guaranteed visually indistinguishable across all Twitter/X UI variants.
- **Context awareness**: Replies do not use original tweet text. They are generic hostile/noisy replies seeded by tweet ID.
- **Selector fragility**: Twitter/X DOM changes may require updates to `utils/selectors.ts`.
- **Testing**: There are no automated unit, DOM, or browser-extension tests yet.
- **Lint/formatting**: No lint or formatter configuration has been added yet.
- **Store packaging**: WXT zip output works, but the old store scripts/assets are archived in `.aside/store/` and may need a new publishing workflow.
- **Firefox**: WXT supports Firefox builds, but Firefox runtime behavior has not been validated.

## Open questions

1. **Keyword awareness**: Should the yapper engine extract keywords from the original tweet text to make some replies contextually relevant (e.g., if you tweet about cooking, a reply says "your pasta takes are criminal")? This increases believability but adds complexity.

2. **Quote tweet simulation**: Should the extension also simulate fake quote tweets (visible as notifications or on the user's tweet)? This is a different injection surface and significantly more complex.

3. **Graduation metric**: Should the extension track how long the user has been using it and surface a "you've been yapped for 30 days" milestone? Could reinforce the training metaphor.

4. **Selector update mechanism**: When Twitter/X changes its DOM, users will need an extension update for `utils/selectors.ts`. Options include manual update via extension release, community-maintained selector patches, or a lightweight hosted selector config, though hosted config conflicts with the offline/no-network principle.

5. **Thread-view strategy**: Should future work prioritize full fake reply cells in thread views, compact previews, or a separate overlay that visually mimics the reply list without changing scroll height?
